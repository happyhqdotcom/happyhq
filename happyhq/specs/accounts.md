# Accounts

How Q identifies users and manages sessions.

## Purpose

Define the user identity layer: email-based signup/login via InstantDB, persistent sessions, and user-to-billing relationships. Accounts answer "who are you?" so that billing can answer "what can you do?"

Accounts are **core product code**, not EE. The dependency flows one direction: EE (billing) depends on accounts for identity. Accounts never import from EE.

Accounts are **optional**. A solo user running Q locally with their own API key never sees a login screen. Accounts activate when the deployment needs identity — primarily when billing is involved.

For deployment-level access control (password gate), see [Auth](auth.md). For Anthropic API credentials, see [Auth](auth.md). For pricing, payments, and usage limits, see [Billing](billing.md).

## When Accounts Activate

| Scenario                          | Accounts? | Why                                                     |
| --------------------------------- | --------- | ------------------------------------------------------- |
| Local + BYO API key               | No        | Solo user, no billing. Zero friction.                   |
| Fly VM demo (password gate)       | No        | Quick throwaway demos. Password gate is enough.         |
| Fly VM / Electron (paid, our key) | Yes       | We're footing the AI bill — need to know who to charge. |
| Future hosted SaaS                | Yes       | Multi-user product with billing.                        |

The rule: **if Q is using HappyHQ's Anthropic key, an account is required.** If the user brings their own key, accounts are unnecessary.

## Auth Layers

Q has up to four independent auth layers. Each is optional:

| Layer                     | What it answers                 | Technology                                   | Spec                  |
| ------------------------- | ------------------------------- | -------------------------------------------- | --------------------- |
| **Password gate**         | "Can you access this instance?" | `Q_PASSWORD` env var + HMAC cookie           | [Auth](auth.md)       |
| **Accounts** (this spec)  | "Who are you?"                  | InstantDB auth (magic code / Google OAuth)   | This spec             |
| **Anthropic credentials** | "Can Q call the AI?"            | API key / OAuth token / proxy / CLI fallback | [Auth](auth.md)       |
| **Billing**               | "What can you do?"              | Stripe subscriptions + InstantDB usage       | [Billing](billing.md) |

The password gate is **temporary** — it exists for quick demo VMs during early user interviews. It will be phased out as accounts become the standard access control. When both are active, password gate runs first (instance access), then accounts (identity).

## Auth Flow

### First-Time User

1. User lands on Q — sees a simple prompt: "Enter your email to get started"
2. User enters email, clicks "Continue" → InstantDB sends a 6-digit magic code to their email
3. User enters the code in the OTP input → account created, session established
4. User lands on the home page

No password to set, no credit card, no friction. Google OAuth is an alternative to magic codes.

### Returning User

1. User lands on Q — InstantDB session cookie is still valid
2. Automatically authenticated — lands on home page
3. If session expired — sees email prompt again, enters a new magic code

### Session Management

InstantDB's client SDK handles session tokens automatically:

- Session persists in the browser (managed by InstantDB SDK)
- Sessions survive page refreshes and browser restarts
- Session expiry configured in InstantDB dashboard
- On expiry, user sees the login prompt — no error state, just re-auth

**Token-based server-side auth**: API routes that need user identity accept the token from the client via `Authorization: Bearer <token>` header. The token comes from `db.useAuth().user.refresh_token` on the client side, and is verified server-side via `adminDb.auth.verifyToken()` in `verifyToken()` from `auth.server.ts`. This follows InstantDB's recommended pattern from their stripe-credits example.

**Common mistake — cookie-based auth does not work**: InstantDB manages its own session tokens via the client SDK. There is no server-side session cookie. Do not attempt cookie sync (e.g. via a `/api/instant` route) — it breaks when auth is client-side. All server-side identity checks must use the token-based pattern above. Server actions that check billing limits MUST fail closed: if `verifyToken()` returns null, return 401 immediately. Never silently bypass billing checks on auth failure (fail-open).

## InstantDB Schema

```
$users (InstantDB built-in users table — note the $ prefix)
  - id: string (auto-generated by InstantDB)
  - email: string (unique, indexed)
  - name: string (optional, from Google OAuth or user-set)
  - avatarUrl: string (optional, from Google OAuth)
  - createdAt: date (stored as epoch ms via Date.now())
  - stripeCustomerId: string (optional, added by billing on first upgrade)
```

The schema is defined in `lib/accounts/schema.ts` using `i.schema()` from `@instantdb/core`. This file is shared infrastructure — importable by both the client SDK (`@instantdb/react`) and admin SDK (`@instantdb/admin`). The `i` helper is imported from `@instantdb/core` (not `@instantdb/react`) to avoid pulling React into server code.

Billing extends this same schema file with `subscriptions`, `usage`, and `taskRuns` entities plus their link definitions (see [Billing](billing.md)).

## Login UI

A simple, centered page — similar in layout to the existing password login page.

### Magic Code Flow

1. Email input + "Continue" button
2. On submit: `db.auth.sendMagicCode({ email })` → UI transitions to 6-digit OTP input
3. User enters code → `db.auth.signInWithMagicCode({ email, code })` → authenticated → redirect to `/`
4. Code verification triggers automatically via `onComplete` when all 6 digits are entered

### Google OAuth Flow

Uses branded Google Sign In (via `@react-oauth/google`) instead of InstantDB's redirect-based OAuth. This shows the app's name on the consent screen instead of "InstantDB". The Google button only renders when `NEXT_PUBLIC_GOOGLE_CLIENT_ID` is set.

1. `GoogleOAuthProvider` wraps a `GoogleLogin` button (rendered below email input with "or" divider)
2. User clicks Google button → Google's JS SDK handles the consent flow inline (no full-page redirect)
3. On success, Google returns an ID token (credential)
4. `db.auth.signInWithIdToken({ clientName: 'google-web', idToken: credential, nonce })` — InstantDB verifies the token and creates/signs in the user
5. `db.useAuth()` detects authentication → redirect to `/`

**Key details:**

- `clientName` is `'google-web'` (must match the OAuth client configured in the InstantDB dashboard)
- A nonce is generated per login attempt via `crypto.randomUUID()` and passed to both Google and InstantDB for security
- The `@react-oauth/google` package provides `GoogleOAuthProvider` and `GoogleLogin` components

### UI States

- **Default**: Email input + "Continue with email" button. When `NEXT_PUBLIC_GOOGLE_CLIENT_ID` is set: "or" divider + branded Google Sign In button
- **Code sent**: 6-digit OTP input (`input-otp` component), "Resend code" link, "Use a different email" back button
- **Error**: Inline error message (e.g., "Something went wrong. Try again.")
- **Loading**: Buttons disabled during auth request, "Sending code..." / "Verifying..." text

The `/login` page is split into a server component (`page.tsx`) and a client component (`login-page.tsx`). The server component detects `Q_PASSWORD` and passes `hasPasswordGate` to the client. The client component implements a state machine:

- **No password gate, accounts enabled**: Render `AccountsLogin` directly
- **Password gate + accounts**: Render `PasswordLogin` first. On success, set `passwordCleared = true` → transition to `AccountsLogin`
- **Password gate only**: Render `PasswordLogin`. On success, redirect to `/`

## Route Protection

InstantDB auth is **client-side**. The `(app)` layout includes an `<AuthGuard />` component (`components/features/auth/auth-guard.tsx`) that uses `db.useAuth()` and redirects to `/login` when the user is not authenticated. This follows InstantDB's recommended pattern — auth state drives UI directly, no server-side cookie checks.

The middleware only handles the password gate (`Q_PASSWORD`). It does **not** check InstantDB auth — that's handled client-side by AuthGuard.

API routes that need server-side auth accept a Bearer token from the client and verify it via `verifyToken()` from `auth.server.ts`.

When both password gate and accounts are active, password gate runs first (middleware, instance access), then AuthGuard handles identity (client-side).

## Email Allowlist

For private VM deployments, `ALLOWED_EMAILS` restricts which email addresses can access the instance after signing in. This is a server-side check that runs after InstantDB authentication.

**How it works:**

1. User signs in via magic code or Google OAuth (InstantDB auth succeeds)
2. `AuthGuard` renders `AllowlistGate`, which calls `GET /api/accounts/me` with the user's Bearer token
3. Server-side `requireAuth()` verifies the token via `verifyToken()`, then checks `isEmailAllowed(email)`
4. If the email is not on the allowlist → 403 response → client signs the user out and shows "Access restricted"
5. If the email is allowed (or `ALLOWED_EMAILS` is unset) → 200 → children render normally

**Configuration:**

```
ALLOWED_EMAILS=user1@company.com,user2@company.com
```

- Comma-separated, case-insensitive (stored lowercase)
- When unset, all authenticated emails are allowed
- Parsed by `getAllowedEmails()` and checked by `isEmailAllowed()` in `lib/accounts/config.ts`

**Key files:**

- `lib/accounts/config.ts` — `getAllowedEmails()`, `isEmailAllowed()`
- `lib/accounts/auth.server.ts` — `requireAuth()` (token verification + allowlist check)
- `components/features/auth/auth-guard.tsx` — `AllowlistGate` component
- `app/api/accounts/me/route.ts` — Lightweight auth + allowlist check endpoint

## API Credential Auto-Configuration

When a user is logged in with an account on a hosted/paid deployment:

- `getAuthEnv()` detects the account and auto-configures `ANTHROPIC_BASE_URL` to route through Cloudflare AI Gateway
- The user never sees the `/setup` page — API access just works
- No API key is stored on the user's machine; the key is injected server-side by the gateway

Local/BYO key users continue to use the existing setup flow unchanged. See [Auth](auth.md) and [Billing](billing.md) for details on the proxy architecture.

## Account Management

Settings page with a complete but minimal account section:

- **Display name**: Editable text field (pre-populated from Google OAuth if used)
- **Email**: Read-only display
- **Avatar**: From Google OAuth (no custom upload in v0)
- **Logout**: Clears InstantDB session, redirects to `/login`
- **Delete account**: Removes user from InstantDB, cancels any Stripe subscription, clears session
- **Billing link**: When billing is active, link to billing settings / Stripe Customer Portal

User info (name/email/avatar) also surfaces in the sidebar.

### `useCurrentUser()` Hook

Components that need identity use a `useCurrentUser()` hook:

- Returns `{ user, isLoading, isAuthenticated }` when accounts are active
- Returns `{ user: null, isLoading: false, isAuthenticated: false }` when accounts are off (module-level constant for stable reference)
- Components should gracefully handle both states

**Note**: `useCurrentUser()` only returns `id` and `email` from `db.useAuth()`. Custom fields (`name`, `avatarUrl`, `createdAt`) require a separate `db.useQuery()` on `$users` — this is done in components that need full profile data (e.g., `AccountSettings`, `AccountFooter`).

## Data Isolation

Not in v0. Each deployment is single-tenant (one user or team per VM/machine).

- All streams on the instance are accessible to all authenticated users
- No per-stream permissions or ownership metadata
- Multi-tenancy (isolating streams per user on shared infrastructure) is a future concern for the hosted SaaS phase

## Code Location

```
lib/accounts/
  schema.ts           — Shared schema (entities, links) for both client and admin SDK
  instant.ts          — InstantDB client SDK initialization (exports db)
  instant.server.ts   — InstantDB admin SDK initialization (exports getAdminDb())
  auth.server.ts      — Server-side auth helpers (verifyToken, requireAuth)
  types.ts            — User type definitions
  config.ts           — Feature flags + allowlist (isAccountsEnabled, getAllowedEmails, isEmailAllowed)
  hooks.ts            — useCurrentUser() React hook (provides user, token)
  actions.server.ts   — Server actions (deleteAccount)
  permissions.json    — InstantDB permission rules

app/api/accounts/delete/route.ts  — Account deletion API route
app/api/accounts/me/route.ts      — Auth + allowlist check (used by AllowlistGate)
```

Accounts live in `lib/accounts/`, not `ee/`. They are core product code. The dependency direction is always: `ee/billing/ → lib/accounts/` (never the reverse).

## Environment Variables

| Variable                       | Required              | Description                                                              |
| ------------------------------ | --------------------- | ------------------------------------------------------------------------ |
| `ACCOUNTS_ENABLED`             | No                    | Set to `true` to activate InstantDB auth (server-side)                   |
| `NEXT_PUBLIC_ACCOUNTS_ENABLED` | When accounts enabled | Client-side accounts flag (needed by login page, hooks)                  |
| `INSTANT_APP_ID`               | When accounts enabled | InstantDB application ID (admin SDK, server-side)                        |
| `NEXT_PUBLIC_INSTANT_APP_ID`   | When accounts enabled | InstantDB app ID for client SDK (Next.js requires `NEXT_PUBLIC_` prefix) |
| `INSTANT_APP_ADMIN_TOKEN`      | When accounts enabled | Admin token for server-side InstantDB operations                         |
| `NEXT_PUBLIC_GOOGLE_CLIENT_ID` | No                    | Google OAuth client ID — enables branded Google Sign In button           |
| `ALLOWED_EMAILS`               | No                    | Comma-separated email allowlist for private deployments (unset = all)    |

`INSTANT_APP_ID` is shared with billing. If billing is enabled, accounts must also be enabled (billing depends on identity).

## Open Source Considerations

Accounts are source-available, part of the open product. The gate test for what requires an account (and by extension, billing):

> "Does this feature cost HappyHQ money or require HappyHQ infrastructure?"

If yes → fair to require an account and billing. If no → should work without an account.

- Streams, tasks, learning, planning, working, templates → all work without accounts
- Using HappyHQ's API key, usage tracking, team collaboration → require accounts

Identity itself is not gated. Billing and usage enforcement are.
